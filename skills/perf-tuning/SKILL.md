# パフォーマンスチューニング

「このAPIが遅い」「この画面が重い」を直すときの手順書です。**計測せずに直さない・1つずつ直す・数字で報告する**が原則です。

## 手順

1. **症状を定量化する** — 「遅い」を数字にする: どの操作が・どんな条件(データ量・同時数)で・何秒/何msか。目標値(例: p95 500ms)を決めてから着手する。
2. **レイヤーを切り分ける** — どこで時間が消えているかを外側から順に測る: ブラウザ(レンダリング/JS)→ ネットワーク(往復数・ペイロードサイズ)→ アプリ(処理時間)→ DB(クエリ時間)。切り分け前にコードを推測で触らない。
3. **ボトルネックを特定する** — レイヤーに応じた計測ツールで犯人を特定する:
   - **DB**: クエリログ + 実行計画(スロークエリログ・`EXPLAIN`)。1リクエストで発行されるクエリの**本数**を必ず数える(N+1 の検出)
   - **API/アプリ**: 処理区間ごとのタイミングログ・プロファイラ・APM。直列になっている外部呼び出し・ループ内 I/O を探す
   - **フロントエンド**: ブラウザ DevTools の Performance/Network パネル、フレームワークのプロファイラ(React DevTools Profiler 等)、Lighthouse。再レンダー回数・バンドルサイズ・画像サイズを見る
4. **1つ直して再計測する** — 最も効く1箇所だけ直し、手順1と同じ条件で再計測する。複数同時に直すと効果の因果が分からなくなる。目標未達なら次のボトルネックへ(直した分だけ順位が入れ替わる)。
5. **回帰を防ぐ** — 効果が出たら: Before/After の数字を PR・作業ログに残す。可能なら閾値つきの性能テスト(応答時間アサーション・クエリ本数アサーション・バンドルサイズ上限)を追加し、監視・アラートの対象にする。

## よくある犯人チェックリスト

- **N+1 クエリ** — 一覧の各行ごとに関連データを1クエリずつ取得(ORM の遅延ロード・ループ内 fetch)。JOIN・eager loading・バッチ取得(IN句/DataLoader)に置き換える
- **インデックス欠落** — WHERE / ORDER BY / JOIN に使う列に対応するインデックスが無い(`EXPLAIN` でフルスキャンを確認)
- **非有界クエリ** — LIMIT なしの全件取得。データが増えた将来に必ず壊れる
- **直列な外部呼び出し** — 依存関係のない複数の I/O を順番に待っている(並列化・バッチ化)
- **ペイロード肥大** — 使わないフィールド・過剰なネストを返している。転送量と直列化コスト両方に効く
- **不要な再レンダー** — 参照が毎回変わる props(インラインオブジェクト/関数)・粒度の粗い state・コンテキストの過剰購読
- **巨大リストの非仮想化** — 数百行超のリストを全 DOM 描画している(仮想化リストに)
- **画像・バンドル** — 未圧縮/実寸より大きい画像、使っていないモジュールのバンドル混入(bundle analyzer で確認)

## ルール

- 計測データなしに「たぶんここが遅い」でコードを変えない。プロファイル・計測結果を先に示す。
- 可読性を犠牲にする最適化は、節約量の数字と理由をコードのすぐ隣に文書化する(計測で正当化できない最適化は入れない)。
- キャッシュの導入は最終手段(無効化の設計とセットでのみ)。まずクエリと構造を直す。
- 報告には必ず Before/After の数字・計測条件・使ったツールを含める。
