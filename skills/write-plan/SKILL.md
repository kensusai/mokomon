# プラン作成(エージェント実行版)

## いつ使うか

`agent-ready plan create`(Claude API 版)を使わずに、このエージェント自身の調査でプランを作るとき。APIキー・追加課金は不要で、出力は API 版と同じフォーマット契約に従うため、`agent-ready plan status` と `/plan-next` がそのまま使える。

## 手順

1. プランの種類を確認する(指定がなければ refactor)。下の「種類別の目的」に従う。
2. リポジトリを**実際に調査する**: ファイル一覧の把握、主要ファイルの読み込み、grep による使用箇所の特定。読んでいないファイルに言及しない。
3. **ベースラインを実測する**: ビルド・テスト・Lint を実際に実行し、結果(成功/失敗件数)を「背景と現状の問題」に記録する。「動いている状態」を暗黙の前提にしない。**壊れている場合は、復旧を「フェーズ 0: ベースライン復旧」として独立タスク群に切り出し**、以降のフェーズは復旧後の状態を前提に書く(復旧の規模が調査で見積もれない場合はその旨をフェーズ 0 に明記する)。
4. `docs/plans/<type>/PLAN.md` に、下のフォーマット契約どおりにプランを書く。既に同名のプランがある場合は上書きせず、更新するか別の type にするかをユーザーに確認する。
5. 完成後、`agent-ready plan status` でチェックボックスが集計されることを確認する。

## フォーマット契約(必須 — plan status / plan-next が解析します)

- 構成: `# タイトル` → 背景と現状の問題(調査で確認したファイル名・根拠を含める) → 依存順の `## フェーズ N: <名前>` → 各フェーズにタスクのチェックリスト → 最後に空の `## 進捗ログ` セクション。
- タスクは必ず `- [ ] <ID>: <内容>` 形式(例: `- [ ] R2-1: ...`)。1タスク = 1コミットで完結し、独立に検証できる粒度。
- 各タスクの下にインデントで: 対象ファイル、手順、検証方法、完了条件を記載。

## 種類別の目的

### refactor
長年蓄積した無駄(重複・デッドコード・責務の混在・過剰な複雑さ・古いパターン)を洗い出し、**挙動を変えない**完全リファクタリングのプランを書く。各フェーズは独立してリリース可能な単位にする。「挙動を変えない」の基準は手順 3 で実測したベースライン(壊れているビルド・テストがあるなら、まずフェーズ 0 で復旧してから基準にする)。

### review
コードベース全体をレビューし、正しさ(バグ・エッジケース・エラー処理・セキュリティ)と品質(重複・複雑さ・テスト不足)の問題を重大度順に洗い出し、修正プランを書く。各指摘には根拠となる `file:line` を含める。

### modernize
依存・ランタイム・パターンの最新化プランを書く。**バージョンは記憶に頼らず実測する**: 依存の最新版は `npm view <pkg> version`(npm)や `pip index versions <pkg>`、ランタイムの EOL は `curl -s https://endoflife.date/api/<product>.json` で確認し、確認できなかったものは「未確認」と明記する。構成: セキュリティ/EOL対応 → パッチ・マイナー更新 → メジャー更新(破壊的変更は1つずつ独立フェーズ) → パターンの近代化。

### それ以外(自由指示)
ユーザーの指示を目的として、同じフォーマット契約でプランを書く。保存先は `docs/plans/<指示に合う短いid>/PLAN.md`。

## ルール

- 調査で確認した事実だけに基づいて書く。推測が混ざる場合は「未確認」と明記する。
- プランを書くだけで、この時点では実装しない。実装は `/plan-next`(`skills/execute-plan/SKILL.md`)で1タスクずつ行う。
