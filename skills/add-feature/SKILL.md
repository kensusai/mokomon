# 機能追加

## いつ使うか

新しい機能や挙動が求められたとき。

## 手順

1. **明確化する。** 要件を言い直し、置いた仮定を列挙する。曖昧な点は黙って推測せず、明示する。
2. **調査する。** 機能が触れるコードを読み、再利用できる既存パターン・ユーティリティを特定する。
3. **計画する。** 各ステップでビルドが通る、小さな作業単位に分割する。
4. **実装する。** 周囲のコードスタイルに従う。差分は機能に集中させる。
5. **テストする。** 新しい挙動をカバーするテストを追加する。
6. **ドキュメント化する。** README など、新しい挙動の影響を受けるドキュメントを更新する。

## ルール

- リファクタリングを機能実装に混ぜない。別途提案する。
- 並行実装を新規に作るより、既存モジュールの拡張を優先する。
- 新しい依存には明示的な理由が必要。

## クリーンアーキテクチャ: レイヤー配置

実装前に、機能の各部分の置き場所を明示してください:

1. **ドメイン** — 新しいエンティティ・値オブジェクト・ビジネスルール。フレームワーク非依存。
2. **ユースケース** — アプリケーションの流れ。ドメイン型とポート(インターフェース)にのみ依存。
3. **インターフェースアダプタ** — ユースケースと外界を変換するコントローラ/プレゼンタ。
4. **インフラ** — ポートの実装(DBアクセス・APIクライアント・UI接続)。

依存は内側にのみ向くように配線する。ドメインやユースケースの中でフレームワークの import が必要に見えたら、代わりにポートを導入する。

## ドキュメント完了条件

機能は以下を満たして初めて完了とする:

- README と `docs/` が新しい挙動を反映し、コピー&ペーストで動くコマンドを載せている。
- 公開エントリポイントに目的・パラメータ・失敗モードを述べる doc コメントがある。
- 自明でない設計判断がトレードオフとともに記録されている。

## パフォーマンス: 機能追加時のチェック

- 一覧 + 関連データを表示する機能は、実装時に**発行クエリ本数を数えて** N+1 が無いことを確認する(テストでクエリ本数をアサートできるならする)。
- 「データが増えたら」を最初から想定する: 上限・ページング・インデックスを、少件数で動く実装の後付けにしない。
- 新しい画面・エンドポイントが既知のホットパスに乗る場合は、変更前後の計測値を PR に含める。

## セキュリティ・バイ・デザイン

- まず機能の信頼境界を定義する: どんな入力が境界を越えるか、誰が呼び出せるか。
- 検証と認可チェックは機能の一部として実装する。後回しのタスクにしない。
- 新しいエンドポイント・外部から到達可能な操作には、少なくとも1つの異常系テスト(禁止アクセス・不正/悪意ある入力)を含める。

入力の種類ごとの定番の落とし穴(該当する機能では必ず対処する):

- **URL を受け取る入力** — プロトコルを許可リスト(`http` / `https`)で検証する。`javascript:` 等をリンクとして出力しない。
- **ユーザーが決める ID・ハンドル名** — 予約語(`admin`・`api`・`login` 等)と既存ルートとの衝突を拒否し、使える文字種と先頭文字を制限する(プロフィール URL がアプリのパスを乗っ取るのを防ぐ)。
- **ファイルアップロード** — 形式(Content-Type とマジックバイト)・サイズ・ファイル名をサーバー側で検証する。
- **更新・削除** — 認可チェックと WHERE 句の絞り込みを必ずセットで実装する(条件漏れの全件更新・IDOR の両方を防ぐ)。
- **重要操作**(退会・メールアドレス変更・支払い情報変更)— 直近の再認証を要求する。
- **ログイン・登録・パスワードリセット** — 応答の文言・時間差でメールアドレスの登録有無を判別させない(列挙防止)。

参考: [Webサービス公開前のチェックリスト(catnose)](https://zenn.dev/catnose99/articles/547cbf57e5ad28)

## シンプルさのゲート

機能を完了と宣言する前に、タスクが要求していないものをすべて取り除く:

- 「あとで使うかも」で追加した未使用のパラメータ・オプション・フック。
- 実装が1つしかない抽象。
- 呼び出しを転送するだけの間接参照。

差分がより小さくても正しくできるなら、より小さくする。

## TDD: Red → Green → Refactor

このプロジェクトは TDD を採用しています。既定の「実装してからテスト」の順序を次に置き換えてください:

1. 既存の挙動と期待される仕様を確認する。
2. 期待される挙動をテストケースとして表現する。
3. テストを実行し、想定した理由で失敗することを確認する。
4. テストを通す最小の実装を書く。
5. スイート全体を実行し、グリーンにする。
6. スイートをグリーンに保ちながらリファクタリングする。
7. ドキュメントを更新する。

## 型ファースト

- 実装の前に型を設計する: 入力・出力・状態・エラーケース。シグネチャの骨組みを先に書き、ロジックを埋める前に型チェッカーに検証させる。
- 新機能の境界にパース/検証を置き、内側は型付きの値だけを扱うようにする。
