# コードベース全体レビュー

## いつ使うか

- 既存プロジェクトの品質をまとめて棚卸ししたいとき(変更差分のレビューは `skills/code-review/SKILL.md`)。
- 過去に作った指摘一覧(`docs/review-findings.md`)を見直し、解消を進めるとき。

## 動作モード

`docs/review-findings.md` の有無で自動的に切り替える:

- **無い**(または全面再レビューを指示された)→ モードA: 全体レビューして指摘一覧を作る
- **有る** → モードB: 未対応の指摘を1件解消する

## モードA: 全体レビューして指摘一覧を作る

1. **範囲を確認する。** 指定がなければソースディレクトリ全体。生成物・vendor・`node_modules` 等は除外する。
2. **機械にかけられるものを先に実行する**。結果は指摘の根拠として使う。ツールで検出できる指摘は1件ずつ列挙せず、「ツール導入・設定修正」として1エントリにまとめる。
3. **2パスで全体を読む。** 第1パス — 正しさ: ロジックの誤り、エッジケース、エラーハンドリング、セキュリティ。第2パス — 品質: 重複、デッドコード、命名、テスト不足、不要な複雑さ。
4. **指摘は現物を読んで確認してから書く。** 呼び出し先まで追って裏を取る。推測の指摘・でっち上げの問題を書かない。指摘ゼロの観点はゼロと報告してよい。
5. **`docs/review-findings.md` に保存する**(形式は後述)。重大度順に並べる。
6. **報告して停止する。** 件数と重大度別の内訳、上位の指摘を要約する。このモードでは修正しない。

## モードB: 指摘を1件解消する

1. `ステータス: 未対応` の指摘を上から1件選ぶ(ユーザー指定があればそれを優先)。
2. 種別に応じた規律で対応する:
   - **バグ**(挙動が変わる)→ `skills/bug-fix/SKILL.md` の再現ファーストで直す。
   - **リファクタ**(挙動を変えない)→ `skills/refactor/SKILL.md` に従う。
   - **設計**(判断が必要)→ 修正せず、選択肢と推奨を提示してユーザーに確認する。
3. 検証まで済んだらステータスを更新する: `対応済み(<変更の1行要約>)` / `見送り(<理由>)`。エントリ本文は書き換えない。
4. **1件で停止して報告する。** 続けて消化する場合はユーザーが再度呼ぶ。

## 指摘一覧の形式(docs/review-findings.md)

```markdown
# コードレビュー指摘一覧

最終レビュー: <YYYY-MM-DD> / 対象: <範囲>

## <連番>. <指摘の要約>
- 重大度: 高|中|低
- 種別: バグ|リファクタ|設計
- 場所: <file:line>
- 問題: <何が悪いか(1〜2行)>
- 根拠: <確認した事実・実行したコマンドと結果>
- 提案: <どう直すか(1〜2行)>
- ステータス: 未対応
```

- 1指摘 = 1エントリ。重大度順に並べる。
- 再レビュー時は既存エントリを消さない: 解消済みはそのまま残し、現物と突き合わせて古くなった未対応エントリはステータスを `失効(<理由>)` に更新し、新しい指摘を追記する。

## レビュー観点

`skills/code-review/SKILL.md` と同じ観点をコードベース全体に適用する。

## パフォーマンスレビューチェックリスト

- ループ内のクエリ・リモート呼び出し(N+1 パターン)はないか。
- ストリーミングやページネーションが使えるのに、非有界のデータをメモリにロードしていないか。
- 一度計算して再利用できる処理を、イテレーションごとに繰り返していないか。
- アクセスパターンに合わないデータ構造(所属判定の線形走査、繰り返しのソート)はないか。
- 無効化の設計がないキャッシュを追加していないか。

## リファクタリング観点のレビュー

- 触れたのに前より散らかったまま残されたファイルを指摘する。
- 挙動変更とリファクタリングが1つのコミット/PRに混在していないか確認する。
- 3回目の出現に達し抽出すべき重複を指摘する。

## セキュリティレビューチェックリスト

すべての差分で追加で確認する:

- **インジェクション**: パラメータ化やコンテキストに応じたエンコードなしに、外部入力が SQL・シェル・HTML・eval に到達していないか。
- **認可**: 権限のないユーザーがこのエンドポイント・操作を呼べないか(IDOR、所有権チェックの欠落)。
- **秘密情報**: コード・ログ・エラーメッセージ・テストフィクスチャに鍵・トークン・認証情報が現れていないか。
- **情報漏えい**: エラーやレスポンスが内部情報(スタックトレース・他ユーザーのデータ・内部ID)を漏らしていないか。
- **危険なデフォルト**: 過剰に許可された CORS、無効化された TLS 検証、有効なままのデバッグエンドポイント/フラグ。

セキュリティの指摘は、差分の大きさにかかわらずブロッカーとして報告する。

## シンプルさレビューチェックリスト

- この差分は、より小さくても正しくできないか。
- 投機的な汎用化(使われていない柔軟性・時期尚早な抽象・誰も頼んでいない設定)はないか。
- 標準ライブラリや既存ユーティリティで済むところに新しい依存を入れていないか。
- 直接呼べば済むところに間接参照(ラッパー・レイヤー・イベント)を挟んでいないか。

## 型安全レビューチェックリスト

- 新しい `any`・キャスト・抑制コメント・弱められたシグネチャはないか。
- 外部データ(API・ストレージ・環境変数)が境界で型付き構造にパースされずに入ってきていないか。
- 判別可能なユニオンにすべき boolean フラグ + オプショナルフィールドの組み合わせはないか。
- 新しいケースの追加を黙って無視してしまう、非網羅的なユニオン/enum の処理はないか。
