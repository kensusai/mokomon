# CLAUDE.md

このファイルは、Claude Code (claude.ai/code) がこのリポジトリで作業する際のガイドです。

## プロジェクト概要

**mokomon** — 6〜7歳向けいきもの育成ゲーム。HTMLプロトタイプ(`prototype/mokomon.html`)で検証済みの仕様を Flutter でストア版として実装する。仕様の正は `docs/game-design.md` とプロトタイプ。

## 技術スタック

- Flutter / Dart(SDK ^3.8.0)。依存は `shared_preferences`(セーブ)と `audioplayers`(効果音)のみ
- 効果音は実行時にWAV合成(`app/lib/audio/`)、フォントは M PLUS Rounded 1c(`app/assets/fonts/`、OFL)

## ディレクトリ構成

- `app/` — Flutter アプリ本体(層構成は `docs/frontend.md` 参照: models=ドメイン / data=マスタ+永続化 / logic=ユースケース / screens・widgets=UI / audio=効果音)
- `docs/` — 設計書(game-design.md が仕様の正)
- `prototype/` — 検証済みHTMLプロトタイプ(仕様の原本)

## 開発コマンド

```bash
cd app
flutter pub get       # 依存取得
flutter run           # 起動(-d macos / -d chrome / 実機)
flutter analyze       # 静的解析
```

## テスト

```bash
cd app && flutter test
```

- ロジック(GameState/GameController/minigames)はテストファースト。UIフローは widget test で固定
- widget test では `pumpAndSettle` を使わない(常時アニメーションがあるため)。固定時間の `pump` を使う

## コーディング規約

- 周囲のコードのスタイル・命名・パターンに合わせる。
- 明示的に依頼されない限り、広範なリファクタリングより小さく焦点を絞った変更を優先する。

## 開発原則

このプロジェクトでは以下の開発原則を採用しています。提案ではなく、すべての変更に適用される拘束力のあるルールです。`skills/` に対応する手順書(playbook)があります。

### クリーンアーキテクチャ

- コードをレイヤーに分離する: ドメイン(エンティティ・ビジネスルール)、ユースケース(アプリケーションロジック)、インターフェースアダプタ(コントローラ・プレゼンタ・ゲートウェイ)、インフラ(フレームワーク・DB・外部API・UI)。
- 依存は内側にのみ向ける: ドメインは何にも依存しない。ユースケースはドメインに依存する。アダプタとインフラはユースケースとドメインに依存する。外向きの import は禁止。
- ドメイン・ユースケースのコードからフレームワーク・DB・HTTP モジュールを import しない。内側の層にインターフェース(ポート)を定義し、インフラ層で実装する。
- レイヤー境界はプレーンなデータ構造(DTO)で越える。フレームワークのオブジェクト(リクエストオブジェクト・ORMエンティティ)を渡さない。
- コードを追加する際は、どのレイヤーに属するかを明言する。既存ファイルがレイヤーを混在させている場合は、拡張する前に分割する。

### ドキュメント重視

- すべての挙動変更は、同じ変更の中でドキュメント更新を伴う — README・`docs/`・関連するエージェントファイル(`CLAUDE.md` / `AGENTS.md`)。
- 公開する関数・モジュール・API には、目的・パラメータ・失敗モードを述べる doc コメントを付ける。コメントは「何を」ではなく「なぜ」を説明する。
- 例は実行可能に保つ: ドキュメント内のコマンドとコードスニペットはコピー&ペーストで動き、現在のコードに対して検証済みであること。
- 新しいモジュールや自明でない機能には、短い「どう動くか」のメモ(設計判断とトレードオフ)を `docs/` かモジュールヘッダに残す。
- コードと矛盾するドキュメントを見つけたら、回避するのではなく現在のタスクの一部として修正する。

### パフォーマンス重視

- ホットパスを把握し、最適化の前に計測する: コストを示すプロファイルやベンチマークなしの投機的なマイクロ最適化はしない。
- すべての変更で古典的な計算量の罠を避ける: N+1 クエリ、非有界データへのネストしたループ、ループ内で繰り返される巻き上げ・キャッシュ可能な処理。
- アクセスパターンに合わせてデータ構造を選ぶ: 所属判定は配列走査より Map、全件ロードよりページネーション/ストリーミング。
- タイトなループの中にアロケーションと I/O を置かない。API が許すなら外部呼び出しをバッチ化する。
- 既知のホットパスに触れるときは、想定するレイテンシへの影響を明言し、計測/ベンチマークを追加・更新する。
- 最適化が可読性を損なう場合は、なぜ存在し何を節約しているかをすぐ隣に文書化する。説明のない賢さは後で単純化されて消える。

### 継続的リファクタリング

- ボーイスカウトルールを適用する: 触れたファイルはタスクのスコープ内で少しきれいにして返す(命名・デッドコード・重複)。
- リファクタリングのコミットと挙動変更のコミットを分ける。両者を混ぜない。
- 変更しづらいコードに追加する前に、まず変更が容易になるようリファクタリングし、それから容易な変更を行う。
- 重複は3回目の出現で共有関数/モジュールに抽出する(三度目の法則)。
- リファクタリングのステップは小さく、検証しながら進める。
- デッドコードは削除する。「念のため」のコメントアウトはしない(履歴は VCS が覚えている)。

### セキュリティ重視

- すべての外部入力(HTTPパラメータ・フォーム・ファイル・ヘッダ・環境変数・外部APIレスポンス)を信頼しない: 使用前に境界で検証・正規化する。
- SQL は必ずパラメータ化クエリ / プリペアドステートメントで書く。SQL・シェルコマンド・HTML を外部入力との文字列連結で組み立てない。
- 出力は出力先のコンテキスト(HTML・URL・シェル・SQL)に合わせて、出力する時点でエスケープ/エンコードする。
- 秘密情報・トークン・個人情報をログに出さず、コミットしない。秘密情報は環境変数かシークレットマネージャから読む。
- 認証に使う Cookie には `HttpOnly` / `Secure` / `SameSite=Lax` 以上を必ず設定する。
- すべてのエンドポイント・操作でサーバー側の認可を強制する。クライアントが機能を隠していることに頼らない。
- 依存を追加する前にレビューする。セキュリティに関わるコードでは、小さな未監査パッケージより標準ライブラリを優先する。
- 認証・暗号・ファイルアップロード・決済に触れる変更では、検討した悪用ケースを列挙する。

### シンプル実装(YAGNI / KISS)

- タスクが要求していることだけを実装する。投機的な汎用化や、まだ存在しない要求のためのオプション・設定を作らない。
- 賢い解より退屈で明白な解を選ぶ。2つの設計が両方動くなら、概念・ファイル・間接参照が少ない方を選ぶ。
- 具体的な利用箇所が2つ以上存在するまで、新しい抽象(インターフェース・基底クラス・レイヤー・ジェネリクス)を導入しない。
- 関数とモジュールは一息で読める大きさに保つ。深いネストと長い引数リストは単純化のサイン。
- 依存を追加する前に、標準ライブラリと既存のプロジェクトユーティリティで済まないか確認する。
- 別々に進化していく2つのものを結合するより、少しの重複を選ぶ。

### TDD(テスト駆動開発)

- それを要求する失敗するテストなしに実装コードを書かない。Red → Green → Refactor で進める。
- 機能追加時: 先にテストを書き、実行し、想定した理由で失敗することを確認してから実装する。
- バグ修正時: 実装に触れる前に、失敗するテストとしてバグを再現する。リグレッションテストなしの修正は出荷しない。
- 失敗するテストを通すのに必要な最小のコードだけを書く。テストより先回りして実装しない。
- リファクタリングはスイート全体がグリーンの間だけ行い、各ステップ後にスイートを再実行する。
- カバレッジのない領域では、変更前に現在の挙動を固定する最小のリグレッションテストを追加する。

### 型安全重視

- 不正な状態を表現不能にする: boolean フラグ + オプショナルフィールドの組み合わせではなく、判別可能なユニオン/enum で選択肢をモデル化する。
- 言語が提供する最も厳格な型チェックを使い、コンパイルを通すために型エラーを抑制しない。
- 「検証」ではなく「パース」する: 外部データ(APIレスポンス・ストレージ・環境変数・ユーザー入力)は境界で一度だけ型付き構造に変換し、内側には型付きの値だけを渡す。
- 役割や単位を型に刻む — 区別された ID 型、branded / newtype ラッパー — モジュール境界を裸のプリミティブで越えない。
- ユニオンの網羅性をコンパイラに強制させる(never チェック / 網羅的 switch)。ケースの追加が、対応漏れの箇所すべてで大きな音を立てて失敗するようにする。

## 開発ワークフロー

このプロジェクトでは、以下の順序で開発を進めます。

1. **プロジェクト概要** — 何を・誰のために・なぜ作るのかを整理し、AIエージェントの前提知識をつくる(手順: `skills/project-overview/SKILL.md` / 成果物: `docs/project-overview.md`)
2. **ドメイン理解** — 業務用語・登場人物・業務ルールを整理し、ドメイン知識をAIが参照できる形にする(手順: `skills/domain-understanding/SKILL.md` / 成果物: `docs/domain.md`)
3. **アーキテクチャ設計** — レイヤー構成・依存方向・モジュール分割を決め、ADR運用を始める(手順: `skills/architecture-design/SKILL.md` / 成果物: `docs/architecture.md`)
4. **技術選定** — 言語・フレームワーク・DB等の選定理由と不採用案を docs/tech-stack.md に明文化する(手順: `skills/tech-selection/SKILL.md` / 成果物: `docs/tech-stack.md`)
5. **DB設計** — エンティティ・リレーション・migration方針を docs/database.md に整理する(手順: `skills/db-design/SKILL.md` / 成果物: `docs/database.md`)
6. **CI構築** — lint・テスト・ビルドを自動実行するCIパイプラインを構築し、方針を docs/ci.md に整理する(手順: `skills/ci-construction/SKILL.md` / 成果物: `docs/ci.md`)
7. **バックエンド開発** — API設計・エラーハンドリング・認証認可などバックエンド実装ルールに沿って開発する(手順: `skills/backend-development/SKILL.md` / 成果物: `docs/backend.md`)
8. **フロントエンド開発** — 画面設計・コンポーネント・state管理・a11yなどフロントエンド実装ルールに沿って開発する(手順: `skills/frontend-development/SKILL.md` / 成果物: `docs/frontend.md`)
9. **インフラ構築** — IaC(CloudFormation / CDK / Terraform 等)でインフラを構築し、方針を docs/infrastructure.md に整理する(手順: `skills/infra-construction/SKILL.md` / 成果物: `docs/infrastructure.md`)
10. **監視設計** — ログ・メトリクス・アラート・障害対応手順を docs/monitoring.md に整理する(手順: `skills/monitoring-design/SKILL.md` / 成果物: `docs/monitoring.md`)

実装に着手する前に、該当フェーズの docs と `skills/` の手順書を必ず確認してください。成果物ドキュメントが存在しない場合は、該当フェーズの SKILL.md に従って先に作成してください。

## AIエージェントへの注意事項

- 新しいコードを書く前に、関連する既存コードを読む。既存のユーティリティを再利用し、重複実装しない。
- 再利用可能なタスク手順書は `skills/` にあります。該当する SKILL.md があれば従ってください。
- スラッシュコマンドは `.claude/commands/` にあります(`/review`, `/fix-bug`, `/add-feature`, `/test`, `/design`)。
- 挙動を変更したら、同じ変更の中で関連するドキュメントとテストを更新する。
- 要件が曖昧な場合は、黙って推測せず、置いた仮定を明示する。
- **このリポジトリを古いまま放置しない。** 依存ライブラリ・言語/ランタイムのバージョン・古いパターンの最新化は、`docs/plans/modernize/PLAN.md` があればそれに従って `/plan-next` で1タスクずつ進める。無い場合、EOL間近のランタイムや大きく遅れた依存に気づいたら `npx agent-ready modernize` によるプラン生成を提案する。更新は必ず検証とセットで行い、破壊的変更を含むメジャー更新は1つずつ切り戻せる単位で行う。
- **このリポジトリは agent-ready の管理下にある。** 作業の区切りで `npx agent-ready check` を実行し、乖離があれば `npx agent-ready update` で修復する。生成ドキュメント・skills への気づき(誤り・曖昧・不足・あると良いスキル)は、`skills/feedback/SKILL.md` の形式で `docs/agent-ready-feedback.md` に記録する。スキルが無い技術に出会ったら、ローカルスキルを起案して new-skill として記録してよい。

## 禁止事項

- 秘密情報・認証情報・`.env` ファイルをコミットしない。
- lockファイルや生成物を手で編集しない。
- チェックを通すためにテストや Lint ルールを無効化・スキップしない。
- 必要な理由を示さずに新しい依存を追加しない。
